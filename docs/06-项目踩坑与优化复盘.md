# 项目踩坑与优化复盘

> 本文档记录企业级RAG文档问答系统开发过程中的5个核心踩坑点及解决方案，供面试口述使用。

## 📌 踩坑一：初始切片无重叠窗口，问答上下文缺失

### 问题描述

**现象**：早期版本采用固定512字符切片，无重叠窗口。导致文档在句子中间被截断，切片之间上下文割裂，RAG问答时答案断章取义。

**影响**：用户体验差，AI回答不连贯，经常出现上下文丢失问题。

### 根因分析

```java
// ❌ 问题代码：直接按固定长度切分
String chunk = content.substring(i, i + CHUNK_SIZE);
```

**问题**：
1. 句子中间被截断
2. 切片之间无上下文关联
3. 检索时无法获取完整语义

### 解决方案

**引入重叠窗口机制**：

```java
// ✅ 优化代码：带重叠窗口的智能切片
private List<String> splitContent(String content) {
    List<String> chunks = new ArrayList<>();
    int overlap = 120; // 重叠窗口大小
    
    for (int i = 0; i < content.length(); i += CHUNK_SIZE - overlap) {
        int end = Math.min(i + CHUNK_SIZE, content.length());
        
        // 尝试在句子边界处切分
        if (end < content.length()) {
            int boundary = findSentenceBoundary(content, end);
            if (boundary > i + CHUNK_SIZE / 2) {
                end = boundary + 1;
            }
        }
        
        chunks.add(content.substring(i, end).trim());
    }
    return chunks;
}
```

**配置化参数**：

```yaml
rag:
  chunk-size: 512
  chunk-overlap: 120  # 新增重叠窗口配置
```

### 面试口述

> "第一个坑是切片策略。最开始我们按固定长度切片，结果句子被截断，问答时上下文丢失。后来我引入了重叠窗口机制，在切片边界设置120字符的重叠区域，同时优化切分算法，保证在句子边界处切分。这样既保证了上下文的连贯性，又通过重叠区域覆盖了边界信息。"

## 📌 踩坑二：未做阈值过滤，无效向量召回过多

### 问题描述

**现象**：向量检索返回Top5结果，但很多相似度低于0.5，导致AI基于无关内容生成答案，甚至答非所问。

**影响**：RAG系统召回质量差，误导LLM生成错误答案。

### 根因分析

```java
// ❌ 问题代码：未过滤低相似度结果
List<SearchResult> results = chromaDB.search(embedding, 5);
for (SearchResult result : results) {
    // 直接使用，没有相似度判断
    context.append(result.getContent());
}
```

**问题**：
1. 未设置相似度阈值
2. 垃圾进垃圾出，干扰LLM判断

### 解决方案

**动态阈值过滤**：

```java
// ✅ 优化代码：相似度阈值动态配置
@Value("${rag.similarity-threshold:0.75}")
private double similarityThreshold;

List<SearchResult> results = chromaDB.search(embedding, 10); // 多召回一些

List<ReferenceSource> sources = new ArrayList<>();
StringBuilder context = new StringBuilder();

for (SearchResult result : results) {
    // ✅ 过滤低相似度结果
    if (result.getSimilarity() >= similarityThreshold) {
        sources.add(buildSource(result));
        context.append(result.getContent()).append("\n\n");
    }
}

if (sources.isEmpty()) {
    // ✅ 无匹配时返回兜底提示
    return "无法从知识库中找到相关信息";
}
```

**配置化参数**：

```yaml
rag:
  similarity-threshold: 0.75  # 相似度阈值可动态调整
  recall-top-num: 5           # 召回数量
```

### 面试口述

> "第二个坑是向量召回质量控制。最开始我们直接用Top5结果，没有过滤低相似度的内容，导致AI基于无关文档生成答案。后来我实现了动态阈值过滤机制，将相似度阈值设置为0.75，并且这个阈值可以通过配置文件动态调整。低于阈值的召回结果会直接过滤掉，只有达标的结果才会进入Prompt构建环节。"

## 📌 踩坑三：同步调用AI导致接口超时，改为异步线程池

### 问题描述

**现象**：Embedding和LLM调用是同步执行的，单次问答耗时3-5秒，超时投诉频发。

**影响**：接口响应慢，用户体验差，前端频繁超时重试。

### 根因分析

```java
// ❌ 问题代码：同步调用AI
public QAResponse ask(String question) {
    float[] embedding = embeddingService.embed(question);  // 阻塞1-2秒
    String answer = llmService.generate(prompt);             // 阻塞2-3秒
    return QAResponse.builder().answer(answer).build();
}
```

**问题**：
1. 同步阻塞主线程
2. 文档上传和切片串行执行
3. 大文件解析时接口超时

### 解决方案

**异步线程池隔离**：

```java
// ✅ 优化代码：线程池配置
@Configuration
public class ThreadPoolConfig {
    
    @Bean("fileParserThreadPool")
    public ThreadPoolTaskExecutor fileParserThreadPool() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(4);
        executor.setMaxPoolSize(8);
        executor.setQueueCapacity(100);
        executor.setThreadNamePrefix("file-parser-");
        executor.setRejectedExecutionHandler(new CallerRunsPolicy());
        return executor;
    }
    
    @Bean("aiCallThreadPool")
    public ThreadPoolTaskExecutor aiCallThreadPool() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(4);
        executor.setMaxPoolSize(8);
        executor.setQueueCapacity(50);
        executor.setThreadNamePrefix("ai-call-");
        return executor;
    }
}

// ✅ 异步切片处理
@Async("fileParserThreadPool")
public void asyncProcessDocument(Long documentId) {
    // 文件切片、向量化异步执行
    log.info("开始异步处理文档: {}", documentId);
}
```

### 面试口述

> "第三个坑是接口超时问题。早期版本AI调用是同步的，单次问答要3-5秒，用户投诉频繁。后来我重构为异步架构，引入线程池隔离机制。文件解析走fileParserThreadPool，AI调用走aiCallThreadPool，实现了主线程的非阻塞化。同时采用@Async注解实现文档处理的异步化，大文件上传不再阻塞接口响应。"

## 📌 踩坑四：文档删除向量残留，脏数据累加

### 问题描述

**现象**：删除文档后，向量库中的向量仍然存在，导致向量库体积持续膨胀，检索结果包含已删除文档内容。

**影响**：向量库脏数据积累，检索结果不准确，存储资源浪费。

### 根因分析

```java
// ❌ 问题代码：只删数据库记录
public void deleteDocument(Long id) {
    documentMapper.deleteById(id);  // 只删了数据库
    // ❌ 遗漏：向量库数据未清理
}
```

**问题**：
1. 数据库和向量库操作不同步
2. 缺少事务保证
3. 删除操作不完整

### 解决方案

**联动删除机制**：

```java
// ✅ 优化代码：联动删除
@Transactional(rollbackFor = Exception.class)
public void deleteDocument(Long id) {
    RagDocument document = documentMapper.selectById(id);
    
    // 1. 逻辑删除文档记录
    document.setIsDelete(1);
    documentMapper.updateById(document);
    
    // 2. 联动删除向量库数据
    chromaDBClient.deleteByDocumentId(id);
    
    // 3. 逻辑删除切片记录
    chunkMapper.delete(new LambdaQueryWrapper<RagDocumentChunk>()
            .eq(RagDocumentChunk::getDocumentId, id));
    
    // 4. 删除本地文件
    Files.delete(Path.of(document.getStoragePath()));
}
```

**向量库删除接口**：

```java
public void deleteByDocumentId(Long documentId) {
    // 根据documentId删除向量库中的所有相关向量
    chromaCollection.delete(
        Where.filter(Map.of("document_id", documentId))
    );
}
```

### 面试口述

> "第四个坑是数据一致性问题。早期版本删除文档时只删了数据库记录，向量库里的向量没有清理，导致检索结果包含已删除文档的内容。后来我实现了联动删除机制，在删除文档时同时清理向量库数据和本地文件，用@Transactional保证事务一致性。现在每次删除操作都会触发向量库清理，彻底解决了脏数据残留问题。"

## 📌 踩坑五：LLM随意编造答案，未做知识库约束Prompt

### 问题描述

**现象**：用户问一个知识库中没有的问题，LLM基于自己的训练知识编造答案，用户无法判断答案是否可信。

**影响**：RAG系统丧失可信度，可能误导用户。

### 根因分析

```java
// ❌ 问题代码：无约束Prompt
String prompt = "问题：" + question + "\n回答：";
String answer = llmService.generate(prompt);  // LLM随意发挥
```

**问题**：
1. Prompt没有知识库约束
2. LLM自由发挥，编造答案
3. 没有兜底机制

### 解决方案

**约束Prompt模板**：

```java
// ✅ 优化代码：带约束的Prompt模板
private static final String PROMPT_TEMPLATE = """
    你是一个专业的文档问答助手。请根据以下提供的参考文档内容，回答用户的问题。

    参考文档：
    {context}

    请遵循以下规则：
    1. 只根据提供的参考文档内容进行回答，不要添加任何外部知识
    2. 如果参考文档中没有相关内容，请直接说"无法从知识库中找到相关信息"
    3. 回答要简洁明了，不要冗长
    4. 如果用户的问题与参考文档无关，请拒绝回答

    用户问题：
    {question}
    """;

// ✅ 调用时填充上下文
String prompt = PROMPT_TEMPLATE
        .replace("{context}", context.toString())
        .replace("{question}", question);
```

**无匹配时兜底**：

```java
// ✅ 无召回结果时返回明确提示
if (sources.isEmpty()) {
    return QAResponse.builder()
            .answer("无法从知识库中找到相关信息")
            .status("NO_MATCH")
            .build();
}
```

### 面试口述

> "第五个坑是幻觉问题。早期版本LLM会基于自己的知识编造答案，用户无法判断答案是否来自知识库。后来我设计了一套Prompt约束机制，在Prompt中明确要求LLM只能基于提供的文档内容回答，并且添加了兜底逻辑：当知识库中没有相关内容时，直接返回'无法从知识库中找到相关信息'，而不是让LLM随意发挥。这样既保证了答案的可信度，又避免了误导用户。"

## 📌 踩坑六：Maven多模块循环依赖导致编译失败

### 问题描述

**现象**：项目编译时报错`java: 程序包com.enterprise.rag.api.dto不存在`，明明DTO类文件存在，但Maven无法正确解析依赖关系。

**影响**：项目无法编译，IDEA报红，开发进度受阻。

### 根因分析

```xml
<!-- ❌ 问题配置：循环依赖 -->
<!-- rag-api模块依赖rag-application -->
<dependency>
    <groupId>com.enterprise</groupId>
    <artifactId>rag-application</artifactId>
</dependency>

<!-- rag-application模块又依赖rag-api -->
<dependency>
    <groupId>com.enterprise</groupId>
    <artifactId>rag-api</artifactId>
</dependency>
```

**问题**：
1. `rag-api` → `rag-application`（Controller需要调用Service）
2. `rag-application` → `rag-api`（Service需要使用DTO）
3. Maven无法解析这种A依赖B、B又依赖A的循环关系

### 解决方案

**调整DTO位置，打破循环依赖**：

```xml
<!-- ✅ 正确的依赖关系 -->
<!-- rag-api依赖rag-application -->
<dependency>
    <groupId>com.enterprise</groupId>
    <artifactId>rag-application</artifactId>
</dependency>

<!-- rag-application不再依赖rag-api -->
<!-- DTO类移动到rag-application模块 -->
```

**模块职责调整**：

| 模块 | 调整前 | 调整后 |
|:---|:---|:---|
| **rag-api** | Controller + DTO | Controller |
| **rag-application** | Service + Manager | Service + Manager + DTO |

**正确的依赖链路**：

```
rag-api（Controller层）
    ↓ 依赖
rag-application（Service + DTO）
    ↓ 依赖
rag-domain（Entity + Enum）
    ↓ 依赖
rag-infrastructure（DAO + AI）
    ↓ 依赖
rag-common（Util + Config）
```

### 面试口述

> "第六个坑是Maven循环依赖问题。最开始我把DTO放在rag-api模块，Service在rag-application模块需要引用DTO，而Controller在rag-api模块需要调用Service，这就形成了循环依赖：rag-api依赖rag-application，rag-application又依赖rag-api。Maven无法解析这种关系，编译直接报错。后来我把DTO类全部移动到rag-application模块，打破了循环依赖，形成了清晰的单向依赖链路。这个问题让我深刻理解了多模块项目的依赖设计原则：底层模块不能依赖上层模块，依赖关系必须是单向的。"

## 📌 踩坑七：向量维度不匹配（1536 vs 1024）

### 问题描述

**现象**：向量插入失败，ChromaDB报错`Collection expecting embedding with dimension of 1536, got 1024`。明明代码中已经将向量维度设为1024，但向量库中的集合仍然使用1536维度。

**影响**：文档无法正确向量化入库，所有切片向量化失败。

### 根因分析

**问题**：
1. 早期设计采用旧的text-embedding-v1/v2模型，维度为1536
2. 后期切换到text-embedding-v3模型，维度变为1024
3. 配置文件中未同步更新维度参数
4. ChromaDB中已存在的集合仍使用旧维度
5. 代码中多处硬编码了1536维度值

```yaml
# ❌ 问题配置：多处配置不一致
rag:
  dimension: 1536  # 未更新

# ❌ 问题代码：RagConstants.java
public static final int VECTOR_DIMENSION_OPENAI = 1536;  // 未更新
```

**排查过程**：
1. 检查embedding模型文档，确认text-embedding-v3输出1024维
2. 检查代码中所有维度配置点
3. 发现RagProperties、RagConstants、ChromaProperties等多处未同步
4. 检查ChromaDB中已存在集合的维度

### 解决方案

**Step 1：统一配置维度**

```yaml
# ✅ application.yml
rag:
  dimension: 1024  # 统一使用1024维度
```

**Step 2：更新常量类**

```java
// ✅ RagConstants.java
public static final int VECTOR_DIMENSION_OPENAI = 1024;
```

**Step 3：更新属性类默认值**

```java
// ✅ RagProperties.java
private Integer dimension = 1024;
```

**Step 4：创建集合时指定维度**

```java
// ✅ ChromaDBClient.java
public String createCollection(String name) {
    // ...
    request.put("dimension", chromaProperties.getEmbeddingDimension());
    // ...
}
```

**Step 5：自动修复维度不匹配**

```java
// ✅ ChromaDBClient.getCollectionId() - 自动检测并修复
if (dimension != null && dimension != chromaProperties.getEmbeddingDimension()) {
    log.warn("集合维度不匹配: {} vs {}, 将删除旧集合并创建新集合", 
             dimension, chromaProperties.getEmbeddingDimension());
    deleteCollection(collectionName);
    return createCollection(collectionName);
}
```

**Step 6：清理旧集合**

手动删除ChromaDB中所有使用1536维度的旧集合：
```bash
curl -X DELETE "http://localhost:8000/api/v2/tenants/default_tenant/databases/default_database/collections/{collection_id}"
```

### 面试口述

> "第七个坑是向量维度不匹配问题。项目从text-embedding-v1切换到v3后，维度从1536变为1024，但配置没有同步更新，导致向量插入时维度冲突。后来我做了三件事：1) 梳理所有维度配置点，统一改为1024；2) 在ChromaDBClient中增加自动检测和修复机制，当发现集合维度不匹配时自动删除重建；3) 清理了所有旧的1536维集合。这个问题提醒我技术选型变更时要全面梳理配置项，不能遗漏。"

## 📌 踩坑八：RagDocumentChunk实体缺少chunkSize字段赋值

### 问题描述

**现象**：文档上传后切片向量化成功，但数据库插入失败，报错`Field 'chunk_size' doesn't have a default value`。

**影响**：向量已入库但元数据未保存，切片记录不完整。

### 根因分析

```java
// ❌ 问题代码：DocumentServiceImpl.java
RagDocumentChunk chunk = RagDocumentChunk.builder()
        .documentId(documentId)
        .chunkIndex(i)
        .chunkContent(chunkContent)
        // ❌ 遗漏：chunkSize字段未赋值

        .build();
```

**问题**：
1. 数据库表中chunk_size字段设置为NOT NULL但无默认值
2. 代码中构建实体时遗漏了该字段
3. MyBatis-Plus的逻辑删除自动填充了部分字段，但chunk_size需要手动设置

### 解决方案

**Step 1：确认表结构**

```sql
-- rag_document_chunk表chunk_size字段定义
chunk_size INT NOT NULL COMMENT '切片长度（字符数）'
```

**Step 2：在实体构建时赋值**

```java
// ✅ DocumentServiceImpl.java
RagDocumentChunk chunk = RagDocumentChunk.builder()
        .documentId(documentId)
        .chunkIndex(i)
        .chunkContent(chunkContent)
        .chunkSize(chunkContent.length())  // ✅ 添加chunkSize字段
        .vectorId(vectorId)
```

### 面试口述

> "第八个坑是实体字段遗漏问题。在构建RagDocumentChunk对象时，漏掉了chunkSize字段的赋值，而这个字段在数据库中是NOT NULL的。这个bug其实很低级，但因为没写单元测试覆盖，所以漏掉了。后来我补上了chunkSize字段的赋值，并且增加了对关键字段的校验。"

## 📌 踩坑九：通义千问API调用异常（URL错误与请求格式）

### 问题描述

**现象**：
1. 调用通义千问API返回404 Not Found
2. 返回错误`url error, please check url!`
3. 使用qwen3.6-flash模型时出现流式输出格式不兼容

**影响**：问答功能无法正常使用，系统只能返回降级响应。

### 根因分析

**问题1：API URL配置错误**

```yaml
# ❌ 问题配置
qwen:
  api-host: https://dashscope.aliyuncs.com/api/v1/services/aigc/text-generation/generation
```

DashScope SDK的base-url不需要包含具体路径，SDK内部会自动拼接。

**问题2：SDK方式vs REST API方式混淆**

项目使用的是SDK方式（DashScope SDK），不应该手动拼接API路径。

**问题3：请求体格式错误**

```java
// ❌ 问题代码：手动构建了错误的请求格式
request.put("model", model);
request.put("input", Map.of("messages", messages));
```

### 解决方案

**方案1：使用OpenAI兼容格式（推荐）**

```java
// ✅ QwenLLMService.java - 重写为OpenAI兼容格式
private static final String OPENAI_COMPATIBLE_URL = 
    "https://dashscope.aliyuncs.com/compatible-mode/v1";

public String generate(String prompt) {
    String apiUrl = OPENAI_COMPATIBLE_URL + "/chat/completions";
    
    // 构建OpenAI格式请求体
    Map<String, Object> requestBody = new HashMap<>();
    requestBody.put("model", "qwen-plus");  // 或 qwen3.6-plus
    requestBody.put("messages", List.of(
        Map.of("role", "user", "content", prompt)
    ));
    requestBody.put("result_format", "message");
    
    HttpHeaders headers = new HttpHeaders();
    headers.setContentType(MediaType.APPLICATION_JSON);
    headers.set("Authorization", "Bearer " + apiKey);
    
    ResponseEntity<Map> response = restTemplate.exchange(
        apiUrl,
        HttpMethod.POST,
        new HttpEntity<>(requestBody, headers),
        Map.class
```

## 📌 踩坑十：RestTemplate超时配置未从配置文件读取

### 问题描述

**现象**：ChromaDB请求超时，报错`I/O error on GET request for "http://localhost:8000/...": Read timed out`。

**影响**：向量库操作频繁超时，文档向量化失败。

### 根因分析

```java
// ❌ 问题代码：RestTemplateConfig.java - 硬编码超时值
@Bean
public RestTemplate restTemplate() {
    SimpleClientHttpRequestFactory factory = new SimpleClientHttpRequestFactory();
    factory.setConnectTimeout(30000);  // ❌ 硬编码
    factory.setReadTimeout(60000);     // ❌ 硬编码
    return new RestTemplate(factory);
}
```

**问题**：
1. 超时值硬编码，没有使用配置文件的值
2. 当配置文件修改时，超时参数不会生效
3. ChromaDB作为本地服务，合理的超时配置很重要

### 解决方案

```java
// ✅ RestTemplateConfig.java - 使用配置属性
@Configuration
public class RestTemplateConfig {
    
    @Autowired
    private ChromaProperties chromaProperties;
    
    @Bean
    public RestTemplate chromaRestTemplate() {
        SimpleClientHttpRequestFactory factory = new SimpleClientHttpRequestFactory();
        factory.setConnectTimeout(chromaProperties.getConnectTimeout());
        factory.setReadTimeout(chromaProperties.getReadTimeout());
        return new RestTemplate(factory);
    }
}
```

```yaml
# ✅ application.yml
chroma:
  host: 0.0.0.0
  port: 8000
  connect-timeout: 30000   # 连接超时30秒
  read-timeout: 60000      # 读取超时60秒
  embedding-dimension: 1024
```

### 面试口述

> "第十个坑是RestTemplate超时配置问题。ChromaDB是本地部署的向量库，偶尔会有响应慢的情况，但RestTemplate的超时值是硬编码的，没有读取配置文件。当ChromaDB处理大批量数据时，容易触发超时。后来我把超时配置改为从ChromaProperties读取，配置文件可以灵活调整。这个问题提醒我，外部依赖的超时配置一定要可配置化，不能写死。"

## 📌 踩坑十一：ChromaDB v2 API使用方式与v1完全不同

### 问题描述

**现象**：
1. 使用v1格式的API端点报404
2. 集合ID格式错误：`Collection ID is not a valid UUIDv4`
3. 向量插入和查询接口路径不对

**影响**：无法正常操作ChromaDB，向量库功能完全不可用。

### 根因分析

**ChromaDB v2 API变化**：

| 方面 | v1 API | v2 API |
|:---|:---|:---|
| 端点格式 | `/api/v1/collections/{id}/...` | `/api/v2/tenants/{tenant}/databases/{database}/collections/{collection_id}/...` |
| 集合ID | 客户端指定字符串 | 服务端自动生成UUIDv4 |
| 集合查询 | `/api/v1/collections` | `/api/v2/tenants/{tenant}/databases/{database}/collections` |
| 创建集合 | 返回客户端指定的ID | 返回服务端生成的UUIDv4 |

```java
// ❌ v1格式（错误）
String url = "/api/v1/collections/rag_documents/add";

// ✅ v2格式（正确）
String url = "/api/v2/tenants/default_tenant/databases/default_database/collections/{collection_id}/add";
```

### 解决方案

```java
// ✅ ChromaDBClient.java - v2 API实现

// 1. 创建集合 - 提取服务端生成的UUID
public String createCollection(String name) {
    // ...
    // 从响应中提取自动生成的collection_id
    String collectionId = response.getId();  // 服务端返回UUIDv4
    return collectionId;
}

// 2. 查询集合 - 使用CRN格式
public String getCollectionId(String name) {
    String url = String.format(
        "%s/api/v2/tenants/%s/databases/%s/collections",
        baseUrl, tenant, database
    );
    // ...
}

// 3. 插入向量 - 使用collection_id（非CRN）
public String insertVector(String collectionId, String content, float[] embedding) {
    String url = String.format(
        "%s/api/v2/tenants/%s/databases/%s/collections/%s/add",
        baseUrl, tenant, database, collectionId
    );
    // ...
}

// 4. 查询向量 - 使用collection_id（非CRN）
public List<Map<String, Object>> query(String collectionId, float[] embedding, int topK) {
    String url = String.format(
        "%s/api/v2/tenants/%s/databases/%s/collections/%s/query",
        baseUrl, tenant, database, collectionId
    );
    // ...
}
```

**CRN格式说明**：
- 完整格式：`{tenant}:{database}:{collection_id}`（仅用于某些特定API）
- 大多数API只需要 `{collection_id}`（服务端生成的UUIDv4）

### 面试口述

> "第十一个坑是ChromaDB v2 API的兼容性问题。ChromaDB升级到v2后，API格式大变样：端点路径变了、集合ID格式变了、连创建流程都不一样了。最坑的是v2会自动生成UUIDv4的collection_id，不再支持客户端自定义。最开始我用v1格式狂报错，后来仔细看了v2文档，重构了整个ChromaDBClient。这个问题告诉我，使用第三方服务时一定要关注版本更新说明，大版本的API变化往往是breaking的。"

## 📌 踩坑十二：Redis缓存导致测试结果不准确

## 📌 踩坑十三：DashScope SDK的ApiException构造器签名问题

### 问题描述

**现象**：编译报错`Cannot resolve constructor 'ApiException(String)'`。

**影响**：代码无法编译，项目无法启动。

### 根因分析

```java
// ❌ 问题代码
throw new ApiException("消息内容");

// ✅ 正确的调用方式
throw new ApiException(new RuntimeException("消息内容"));
```

**原因**：DashScope SDK的ApiException类没有接受String参数的构造器，只有以下构造器：
- `ApiException(Throwable)`
- `ApiException(Status)`

### 解决方案

**统一异常抛出方式**：

```java
// ✅ QwenLLMService.java
} catch (Exception e) {
    throw new ApiException(new RuntimeException("API调用异常: " + e.getMessage(), e));
}
```

**封装工具方法**：

```java
// ✅ 统一异常处理
private ApiException wrapException(String message, Exception e) {
    return new ApiException(new RuntimeException(message, e));
}
```

### 面试口述

> "第十三个坑是SDK API变化导致的不兼容问题。DashScope SDK更新后，ApiException的构造器签名变了，不再支持直接传入String。这其实是SDK使用者经常会遇到的问题——第三方SDK更新后API不兼容。解决办法就是封装一层，统一异常抛出方式，避免散落在各处。"
